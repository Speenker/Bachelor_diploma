# Проверки API 

Маркировка проверок:

- `smoke` — быстрые базовые проверки работоспособности.
- `regression` — стабильный набор по ключевым операциям.
- `functional` — расширенные проверки (негативные сценарии, граничные случаи, доступы).
- `slow` — длительные проверки, которые можно исключать из обычных прогонов.

## Что проверяется

Проверки работают только через REST API. В область входят:

- вход и запрет доступа без авторизации;
- операции по основным сущностям: доски, списки, карточки, дорожки;
- обработка ошибок (в том числе ситуация «HTTP 200 + ошибка в JSON»);
- устойчивость прогонов на локальном стенде (короткие polling’и после delete, повторы delete и корректные `skip` при временной недоступности стенда).

В область не входят UI‑проверки и нагрузочные тесты.

## Запуск

### Автоматизированные прогоны

В папке проекта есть отдельные команды прогона. Они сами переходят в `diploma-api-tests`, поэтому их можно запускать из любой текущей директории.

```powershell
# Регрессия (формирует отчёт JUnit XML)
.\run_regression.ps1
run_regression.cmd

# Функциональные без slow
.\run_functional_no_slow.ps1
run_functional_no_slow.cmd
```

Отчёт регрессии сохраняется в `reports/regression/junit.xml`, краткая сводка — в `reports/regression/summary.txt`.

Аналогично для прогона `functional and not slow`: `reports/functional_no_slow/junit.xml` и `reports/functional_no_slow/summary.txt`.

### Ручной запуск через pytest

```powershell
# Все проверки
pytest -q

# Только быстрые
pytest -m smoke -q

# Регрессия
pytest -m regression -q

# Функциональные
pytest -m functional -q

# Функциональные без slow
pytest -m "functional and not slow" -q
```

## Структура

- `diploma_tests/` — общая библиотека проекта
  - `config.py` — чтение настроек из окружения/`.env`
  - `client.py` — клиент‑обёртка над REST API (CRUD по сущностям)
  - `http_helpers.py` — сетевые повторы и проверка «unauthorized» ответов
  - `waiters.py` — ожидания/polling для подтверждения удаления
- `tests/` — тесты
  - `smoke/` — smoke сценарии, которые быстро проверяют стенд
  - `regression/` — регрессионные проверки по ключевым операциям
  - `functional/` — расширенные проверки (негативные сценарии, контракты ошибок, доступы, устойчивость)


### Smoke 

Файл: `tests/smoke/test_smoke_flow.py`

- успешный вход и проверка, что токен подходит для защищённого запроса;
- запрет доступа без авторизации;
- создание доски, получение дорожек;
- удаление карточки с подтверждением через polling;
- best‑effort очистка данных.

### Regression

Аутентификация и запрет доступа:

- `tests/regression/test_regression_auth.py`
  - вход с неверным/пустым паролем, отсутствие токена;
  - отказ без авторизации и с неверным токеном на защищённых эндпоинтах.

Доски:

- `tests/regression/test_regression_boards.py`
  - создание доски (контракт ответа);
  - видимость доски в списке;
  - чтение доски по id;
  - удаление доски и подтверждение отсутствия.

Списки:

- `tests/regression/test_regression_lists.py`
  - создание списка (возвращается id);
  - видимость списка в `GET /lists`;
  - чтение списка по id;
  - удаление списка и подтверждение отсутствия.

Карточки:

- `tests/regression/test_regression_cards.py`
  - создать → прочитать → обновить → переместить → прочитать → удалить;
  - подтверждение перемещения/удаления через polling;
  - best-effort очистка данных.

Права доступа:

- `tests/regression/test_regression_permissions.py`
  - приватная доска недоступна второму пользователю;
  - add/remove участника доски меняет видимость и доступ;
  - при обнаружении дефекта в ACL сценарии помечаются как `xfail`.

### Functional

Реализовано:

- `tests/functional/test_functional_auth.py`
  - успешный вход (по username/email) и проверка токена;
  - негативные случаи по учётным данным;
  - отказ на защищённом эндпоинте при неверном/пустом `Authorization`;
  - вход вторым пользователем (если настроен).
- `tests/functional/test_functional_boards.py`
  - контракт и видимость созданной доски;
  - негативный сценарий на создание с неверным набором полей;
  - проверка, что защищённые эндпоинты требуют авторизацию;
  - проверки доступа между пользователями (если настроены два пользователя);
  - добавление/удаление участника доски (если доступно на стенде);
  - запрос смены роли участника;
  - повторное удаление доски.
- `tests/functional/test_functional_lists.py`
  - проверки авторизации для эндпоинтов списков;
  - контракт: создать → получить → удалить;
  - негативный сценарий: создание списка без title;
  - проверки доступа ко спискам вторым пользователем (если настроен).

- `tests/functional/test_functional_cards.py`
  - контракт: создать → получить → обновить → переместить → удалить;
  - проверки авторизации на карточках;
  - негативные случаи по входным данным (могут `skip`, если стенд принимает некорректные значения).

- `tests/functional/test_functional_error_contracts.py`
  - контракты ошибок для «невалидный JSON» и «отсутствует Content-Type»;
  - допускает варианты ответа Wekan (HTTP error или HTTP 200 + error object).

- `tests/functional/test_functional_validation.py`
  - валидация входных данных для списков/карточек;
  - проверки могут `skip`, если стенд не валидирует сценарий.

- `tests/functional/test_functional_permissions.py`
  - запрет доступа второму пользователю к приватной доске (если права enforced);
  - выдача доступа к доске второму пользователю через members add/remove (если права enforced).

- `tests/functional/test_functional_business_flows.py`
  - бизнес‑сценарий: создать → обновить → переместить → удалить карточку;
  - сценарий с двумя досками: сущности не «смешиваются» между досками;
  - bulk: создать пачку карточек и подтвердить видимость.

- `tests/functional/test_functional_resilience.py` (`slow`)
  - повторные удаления (ожидаемое поведение и отсутствие падений);
  - устойчивость к кратковременной недоступности стенда (корректные `skip`);
  - независимость от порядка cleanup (удаление в любом порядке не приводит к падениям).

Примечания:

- Если стенд временами недоступен (например, `WinError 10061/10053`), часть тестов может быть корректно `skip`.
- Для `GET/PUT/DELETE` включены ограниченные повторы; для `POST` используются recovery‑паттерны (по title)

## Матрица покрытия

Пояснения:

- `skip` — корректное поведение при ограничениях/флаках окружения (например, стенд временно недоступен или не настроен второй пользователь).
- `xfail` — обнаруженный дефект/несоответствие поведения продукта (например, проблемы с ACL/правами).
- ретраи включены только для безопасных методов `GET/PUT/DELETE`; для `POST` вместо повторов используется recovery по title там, где это возможно.

## Что не покрыто

Wekan API шире текущего набора. Сейчас не покрыты (или покрыты минимально):

- работа с вложениями (attachments), чеклистами (checklists), метками/лейблами, комментариями;
- кастомные поля, шаблоны, активности/аудит, нотификации, интеграции и webhooks;
- управление пользователями/ролями на уровне админа;
- импорт/экспорт, backup/restore;
- UI и end-to-end сценарии через браузер.

## Ядро регрессии (15 сценариев)

Эти сценарии используются как обязательный минимум регрессии.

1) Успешный вход пользователя (`POST /users/login`).
2) Вход с неверным паролем (`POST /users/login`).
3) Запрет доступа без авторизации.
4) Создание доски (`POST /api/boards`).
5) Удаление доски (`DELETE /api/boards/{boardId}`).
6) Видимость доски в списке досок (`GET /api/users/{userId}/boards`).
7) Дорожки по умолчанию (`GET /api/boards/{boardId}/swimlanes`).
8) Создание списка (`POST /api/boards/{boardId}/lists`).
9) Видимость списка (`GET /api/boards/{boardId}/lists`).
10) Удаление списка (`DELETE /api/boards/{boardId}/lists/{listId}`).
11) Создание карточки (`POST /api/boards/{boardId}/lists/{listId}/cards`).
12) Видимость карточки по дорожке (`GET /api/boards/{boardId}/swimlanes/{swimlaneId}/cards`).
13) Обновление карточки (`PUT /api/boards/{boardId}/lists/{listId}/cards/{cardId}`).
14) Перемещение карточки (`PUT /api/boards/{boardId}/lists/{fromListId}/cards/{cardId}` + новый `listId`).
15) Удаление карточки (`DELETE /api/boards/{boardId}/lists/{listId}/cards/{cardId}`).

Дополнительно (если настроены 2 пользователя):

16) Приватная доска недоступна второму пользователю (ожидаемое поведение; при дефекте — `xfail`).
17) Add/remove участника доски меняет видимость и доступ (при дефекте — `xfail`).


# План разработки дипломного набора автотестов API

## Назначение документа

Настоящий документ фиксирует объём, структуру и содержание дипломного набора автоматизированных тестов для серверной части сервиса управления задачами на примере Wekan.
План предназначен для последовательной реализации тестов и для предоставления на проверку как часть выпускной квалификационной работы.

## Объект и границы тестирования

Объект тестирования: REST API сервиса управления задачами Wekan.

В границы тестирования входит:
- проверка аутентификации и авторизации;
- операции управления основными сущностями предметной области: доски, списки, карточки, дорожки;
- проверка валидации входных данных и корректности обработки ошибок;
- проверка доступов и ограничений при наличии двух пользователей;
- проверка устойчивости тестового набора к нестабильности среды и повторяемости результатов.

В границы тестирования не входит:
- тестирование пользовательского интерфейса;
- нагрузочное тестирование и измерение производительности в строгом смысле;
- тестирование внешних интеграций, не представленных в целевом стенде.

## Термины и классификация наборов

В проекте используются три уровня тестовых наборов.

### Smoke

Минимальный быстрый набор проверок, который отвечает на вопрос, работоспособен ли сервис на базовом уровне.

Назначение:
- быстро выявлять критические отказа инфраструктуры и базовых функций;
- использовать как первый этап любого прогона.

Требования:
- выполнение не более 1–2 минут;
- минимальная зависимость от окружения;
- высокая устойчивость к флакам.

### Regression

Стабильный регрессионный набор проверок, который запускается регулярно после изменений, чтобы подтвердить, что ранее работавшие сценарии не нарушены.

Назначение:
- фиксировать контракт ключевых функций;
- обеспечивать проверку типовых бизнес‑операций по основным сущностям.

Требования:
- выполнение обычно не более 5–10 минут на локальном стенде;
- независимость тестов друг от друга;
- обязательная очистка тестовых данных по завершении.

### Functional

Расширенный функциональный набор, который покрывает более широкий спектр требований: негативные проверки, граничные условия, контракты ошибок, сценарии доступа, вариативность входных данных.

Назначение:
- предоставить доказательную базу глубины тестирования;
- показать системный подход к качеству API.

Требования:
- допускается более длительное выполнение;
- допускается маркировка части сценариев как медленных.

## Архитектура тестового проекта

Тестовый проект организуется следующим образом.

- `diploma_tests/` содержит библиотечный код проекта
	- `config.py` читает настройки окружения
	- `client.py` содержит клиент‑обёртку для REST API
- `tests/` содержит тесты
	- `tests/smoke/` минимальный smoke набор
	- `tests/regression/` регрессионный набор
	- `tests/functional/` расширенный функциональный набор
	- `tests/conftest.py` содержит фикстуры pytest
- `TEST_SUITE.md` содержит ядро регрессии в виде перечня сценариев с привязкой к эндпоинтам

## Целевые показатели по объёму

Целевые показатели выбраны так, чтобы результат выглядел полноценно и убедительно для дипломной работы, но оставался реалистичным по срокам реализации.

- Smoke: 10–12 тестов
- Regression: 30–40 тестов
- Functional: 60–90 тестов

Допускается изменение целевых величин в пределах ±10 процентов при наличии аргументации.

## Требования к общим принципам тестирования

### Изоляция и данные

- Каждый тест обязан создавать собственные тестовые данные с уникальными идентификаторами.
- Тесты не должны зависеть от порядка выполнения.
- По возможности применяется best‑effort cleanup в `finally` или через фикстуры.
- Стенд должен содержать заранее созданного пользователя для входа, так как саморегистрация через API может быть отключена.

### Устойчивость и повторяемость

- Для HTTP запросов используются сессии с таймаутами и повторами при сетевых сбоях.
- Учитывается особенность Wekan: в отдельных случаях ошибка может быть отражена в JSON‑теле даже при успешном HTTP статусе.
- В проверках негативных сценариев авторизации допускается вариативность статус‑кодов, однако запрещено принимать успешную выдачу токена.

### Отчётность

Минимальная отчётность:
- выгрузка JUnit XML отчёта в каталог `reports/`.

Рекомендуемая отчётность:
- Allure отчёт, включающий шаги и вложения с урезанным логом HTTP взаимодействий без секретов.

## Набор Smoke

### Цель

Подтвердить доступность сервиса и работоспособность ключевых операций CRUD в минимальном объёме.

### Целевой объём

10–12 тестов.

### Перечень сценариев

Ниже приведён рекомендуемый состав smoke набора. В рамках реализации допускается объединение нескольких пунктов в один end‑to‑end тест при условии сохранения покрытия, но в отчётности рекомендуется сохранять 10–12 отдельных тестов для демонстрации структуры набора.

1. Успешный вход пользователя
	 - Эндпоинт: `POST /users/login`
	 - Проверка: присутствует `token`, токен применим для авторизованного запроса

2. Запрет доступа без авторизации
	 - Эндпоинт: любой защищённый, например `GET /api/users/:id/boards`
	 - Проверка: отказ в доступе либо ошибка в теле ответа

3. Создание доски
	 - Эндпоинт: `POST /api/boards`
	 - Проверка: возвращаются `_id` и `defaultSwimlaneId`

4. Получение дорожек доски
	 - Эндпоинт: `GET /api/boards/:boardId/swimlanes`
	 - Проверка: список не пустой

5. Создание списка
	 - Эндпоинт: `POST /api/boards/:boardId/lists`
	 - Проверка: возвращается `_id`

6. Получение списков и наличие созданного
	 - Эндпоинт: `GET /api/boards/:boardId/lists`
	 - Проверка: созданный список присутствует

7. Создание карточки
	 - Эндпоинт: `POST /api/boards/:boardId/lists/:listId/cards`
	 - Проверка: возвращается `_id`

8. Получение карточек по дорожке и наличие созданной
	 - Эндпоинт: `GET /api/boards/:boardId/swimlanes/:swimlaneId/cards`
	 - Проверка: созданная карточка присутствует

9. Обновление карточки по заголовку
	 - Эндпоинт: `PUT /api/boards/:boardId/lists/:listId/cards/:cardId`
	 - Проверка: изменения сохраняются

10. Перемещение карточки в другой список
	 - Предусловие: создан второй список
	 - Эндпоинт: `PUT /api/boards/:boardId/lists/:fromListId/cards/:cardId` с новым `listId`
	 - Проверка: карточка видна в целевом списке или в выдаче по дорожке с обновлённым `listId`

11. Удаление карточки
	 - Эндпоинт: `DELETE /api/boards/:boardId/lists/:listId/cards/:cardId`
	 - Проверка: удаление подтверждено

12. Завершение и очистка данных
	 - Эндпоинты: удаление списка и доски
	 - Проверка: отсутствие ошибок очистки

## Набор Regression

### Цель

Зафиксировать поведение ключевых функций и обеспечить регулярную проверку после изменений.

### Целевой объём

30–40 тестов.

### Базовое ядро

В основу регрессионного набора включается ядро из `TEST_SUITE.md`, которое покрывает 15 сценариев по аутентификации, доскам, спискам и карточкам.

### Расширение регрессионного набора

Для достижения целевого объёма рекомендуется добавить следующие группы проверок.

#### Аутентификация и авторизация

Целевой объём: 6–8 тестов.

- Успешный вход пользователя
	- `POST /users/login`
- Неверный пароль и отсутствие токена
	- `POST /users/login`
- Валидация пустых значений учётных данных
	- `POST /users/login`
- Запрет доступа без авторизации для нескольких эндпоинтов
	- параметризация по `GET /api/users/:id/boards`, `GET /api/boards/:boardId/lists`, `GET /api/boards/:boardId/swimlanes`
- Запрет доступа при некорректном токене
	- любой защищённый эндпоинт

#### Доски

Целевой объём: 8–10 тестов.

- Создание доски
	- `POST /api/boards`
- Удаление доски
	- `DELETE /api/boards/:boardId`
- Проверка видимости доски в списке досок пользователя
	- `GET /api/users/:id/boards`
- Создание двух досок и проверка наличия обеих
	- `POST /api/boards`, `GET /api/users/:id/boards`
- Негативная проверка удаления несуществующей доски
	- `DELETE /api/boards/:boardId`
- Негативная проверка создания доски с некорректным названием
	- `POST /api/boards`
- Повторное удаление уже удалённой доски
	- `DELETE /api/boards/:boardId`

#### Дорожки и списки

Целевой объём: 8–10 тестов.

- Наличие дорожек по умолчанию
	- `GET /api/boards/:boardId/swimlanes`
- Создание списка
	- `POST /api/boards/:boardId/lists`
- Проверка видимости списка
	- `GET /api/boards/:boardId/lists`
- Удаление списка
	- `DELETE /api/boards/:boardId/lists/:listId`
- Создание двух списков и проверка наличия обоих
	- `POST /api/boards/:boardId/lists`, `GET /api/boards/:boardId/lists`
- Негативная проверка создания списка для несуществующей доски
	- `POST /api/boards/:boardId/lists`
- Негативная проверка удаления несуществующего списка
	- `DELETE /api/boards/:boardId/lists/:listId`

#### Карточки

Целевой объём: 10–14 тестов.

- Создание карточки с минимально корректным набором полей
	- `POST /api/boards/:boardId/lists/:listId/cards`
- Получение карточек по дорожке
	- `GET /api/boards/:boardId/swimlanes/:swimlaneId/cards`
- Обновление заголовка карточки
	- `PUT /api/boards/:boardId/lists/:listId/cards/:cardId`
- Обновление описания карточки
	- `PUT /api/boards/:boardId/lists/:listId/cards/:cardId`
- Обновление нескольких полей карточки
	- `PUT /api/boards/:boardId/lists/:listId/cards/:cardId`
- Перемещение карточки между списками
	- `PUT /api/boards/:boardId/lists/:fromListId/cards/:cardId` с новым `listId`
- Удаление карточки
	- `DELETE /api/boards/:boardId/lists/:listId/cards/:cardId`
- Негативная проверка создания карточки без `authorId`
	- `POST /api/boards/:boardId/lists/:listId/cards`
- Негативная проверка создания карточки без `swimlaneId`
	- `POST /api/boards/:boardId/lists/:listId/cards`
- Негативная проверка обновления несуществующей карточки
	- `PUT /api/boards/:boardId/lists/:listId/cards/:cardId`
- Негативная проверка перемещения карточки в несуществующий список
	- `PUT /api/boards/:boardId/lists/:fromListId/cards/:cardId`

## Набор Functional

### Цель

Продемонстрировать широту и глубину тестирования REST API, включая негативные сценарии, граничные условия, контракты ошибок и проверки доступа.

### Целевой объём

60–90 тестов.

### Структура набора и рекомендуемый состав

#### Валидация входных данных

Целевой объём: 15–25 тестов.

Рекомендуемые проверки:
- пустые значения обязательных полей
- строковые поля максимальной длины
- строковые поля с Unicode символами
- значения с ведущими и завершающими пробелами
- некорректные идентификаторы в пути запроса

Применяется параметризация наборов данных для сокращения дублирования кода тестов.

#### Контракты ошибок и предсказуемость ответов

Целевой объём: 10–15 тестов.

Рекомендуемые проверки:
- наличие диагностических полей в ответах об ошибках
- согласованность формата ошибок в пределах одной группы эндпоинтов
- обработка некорректного JSON тела запроса
- обработка отсутствующего или неверного заголовка `Content-Type`
- учёт случаев, когда ошибка присутствует в JSON при успешном HTTP статусе

#### Права доступа на уровне пользователей

Целевой объём: 8–15 тестов.

Условие: наличие двух тестовых пользователей.

Рекомендуемые проверки:
- пользователь B не имеет доступа к данным пользователя A по умолчанию
- пользователь B не может удалять или изменять сущности пользователя A
- при предоставлении доступа к доске проверяется появление доступа на чтение и на изменение

Если управление участниками доски через API затруднено, допускается оформить предоставление доступа как предусловие стенда.

#### Бизнес‑сценарии на уровне предметной области

Целевой объём: 10–20 тестов.

Рекомендуемые сценарии:
- создание задачи, уточнение описания, перемещение по стадиям, удаление
- одновременная работа с двумя досками и несколькими списками
- массовое создание карточек и проверка их видимости

#### Устойчивость и повторяемость результатов

Целевой объём: 5–10 тестов.

Рекомендуемые проверки:
- повторные запросы удаления
- корректная реакция на временные сетевые сбои без падения набора
- отсутствие зависимости от порядка выполнения

## Отчётность и автоматизация запуска

### Режимы запуска

- все тесты: `pytest -q`
- smoke: `pytest -m smoke -q`
- regression: `pytest -m regression -q`
- функциональные: `pytest -m functional -q`

### Артефакты

Рекомендуемые артефакты дипломного прогона:
- JUnit XML отчёт
- Allure отчёт
- лог прогона

### Непрерывная интеграция

Рекомендуется настроить выполнение тестов в CI.

Минимальный состав:
- job для smoke набора
- job для regression набора
- сохранение отчётов как артефактов

При отсутствии возможности запускать стенд в CI допускается выполнять прогоны локально и прикладывать отчёты к материалам диплома.

## План реализации по этапам

Этап 1. Фиксация ядра
- реализовать полный набор из `TEST_SUITE.md`
- убедиться в стабильности: два последовательных прогона regression без флаков

Этап 2. Расширение регрессии
- довести регрессионный набор до 30–40 тестов
- унифицировать клиентские методы в `diploma_tests/client.py`

Этап 3. Реализация функционального набора
- реализовать группы валидации и контрактов ошибок
- добавить проверки доступа при наличии двух пользователей
- оформить отдельные бизнес‑сценарии

Этап 4. Отчётность и воспроизводимость
- включить JUnit XML отчёт
- добавить Allure отчёт
- оформить примеры команд запуска и типовой отчёт дипломного прогона

## Критерии готовности

Набор считается готовым для диплома при выполнении следующих условий.

- реализованы целевые объёмы smoke, regression и functional наборов либо представлены обоснованные отклонения
- тесты воспроизводимы и независимы
- тестовые данные корректно очищаются
- присутствует отчётность, позволяющая подтвердить результаты прогонов
- описаны принципы устойчивости к сетевым сбоям и особенности API

# Набор сценариев ядра регрессионного тестирования

## Назначение документа

Настоящий документ описывает ядро регрессионного набора автоматизированных тестов REST API сервиса управления задачами на примере Wekan.
Документ используется как список обязательных сценариев, которые должны быть реализованы в коде тестов и подтверждены результатами прогонов.

Полный мастер‑план по структуре дипломного набора и целевым объёмам тестов приведён в `plan.md`.

## Определения

Smoke набор представляет собой минимальный набор проверок базовой работоспособности сервиса и является первым уровнем контроля качества.

Регрессионный набор представляет собой стабильный набор проверок ключевых функций, запускаемый регулярно для контроля отсутствия непреднамеренных изменений поведения.

## Особенности поведения API

При реализации негативных сценариев необходимо учитывать следующие особенности стенда.

- При ошибках авторизации сервис может возвращать HTTP 401 или 403.
- В отдельных конфигурациях возможна ситуация, когда HTTP статус равен 200, а информация об ошибке присутствует в JSON теле ответа.
- При локальном запуске возможны кратковременные сетевые обрывы соединения. Тестовый набор должен быть устойчив к таким сбоям посредством ограниченных повторов сетевых запросов.

## Ядро регрессии

Ниже приведён перечень из 15 обязательных сценариев, покрывающих ключевые операции по основным сущностям.

### Аутентификация

1. Успешный вход пользователя
- Эндпоинт: `POST /users/login`
- Проверки:
  - в ответе присутствует `token` и идентификатор пользователя
  - токен применим для вызова защищённого эндпоинта

2. Вход пользователя с неверным паролем
- Эндпоинт: `POST /users/login`
- Проверки:
  - запрос отклоняется
  - токен не выдаётся

3. Запрет доступа без авторизации
- Эндпоинт: любой защищённый эндпоинт без заголовка `Authorization: Bearer <token>`
- Проверки:
  - отказ в доступе либо ошибка в теле ответа

### Доски

4. Создание доски
- Эндпоинт: `POST /api/boards`
- Проверки:
  - в ответе присутствует `_id`
  - в ответе присутствует `defaultSwimlaneId`

5. Удаление доски
- Эндпоинт: `DELETE /api/boards/{boardId}`
- Проверки:
  - удаление подтверждено корректным идентификатором

6. Наличие созданной доски в списке досок пользователя
- Эндпоинт: `GET /api/users/{userId}/boards`
- Проверки:
  - созданная доска присутствует в выдаче

### Дорожки и списки

7. Наличие дорожек по умолчанию
- Эндпоинт: `GET /api/boards/{boardId}/swimlanes`
- Проверки:
  - список дорожек не пустой
  - присутствует дорожка по умолчанию, соответствующая `defaultSwimlaneId`

8. Создание списка
- Эндпоинт: `POST /api/boards/{boardId}/lists`
- Проверки:
  - в ответе присутствует `_id`

9. Наличие созданного списка в списках доски
- Эндпоинт: `GET /api/boards/{boardId}/lists`
- Проверки:
  - созданный список присутствует в выдаче

10. Удаление списка
- Эндпоинт: `DELETE /api/boards/{boardId}/lists/{listId}`
- Проверки:
  - удаление подтверждено корректным идентификатором

### Карточки

11. Создание карточки
- Эндпоинт: `POST /api/boards/{boardId}/lists/{listId}/cards`
- Требования к данным:
  - запрос должен включать `authorId` и `swimlaneId`
- Проверки:
  - в ответе присутствует `_id`

12. Получение карточек по дорожке
- Эндпоинт: `GET /api/boards/{boardId}/swimlanes/{swimlaneId}/cards`
- Проверки:
  - созданная карточка присутствует в выдаче

13. Обновление карточки
- Эндпоинт: `PUT /api/boards/{boardId}/lists/{listId}/cards/{cardId}`
- Проверки:
  - изменение заголовка или описания сохраняется

14. Перемещение карточки между списками
- Предусловие: наличие двух списков в рамках одной доски
- Эндпоинт: `PUT /api/boards/{boardId}/lists/{fromListId}/cards/{cardId}` с указанием целевого `listId`
- Проверки:
  - карточка относится к целевому списку

15. Удаление карточки
- Эндпоинт: `DELETE /api/boards/{boardId}/lists/{listId}/cards/{cardId}`
- Проверки:
  - удаление подтверждено корректным идентификатором
  - карточка отсутствует в выдаче по дорожке после удаления

## Привязка к реализации

Реализация сценариев размещается в следующих файлах.

- Smoke и базовые end‑to‑end проверки: `tests/smoke/test_smoke_flow.py`
- Регрессионные проверки: `tests/regression/test_regression_auth.py`, `tests/regression/test_regression_lists.py`

## Запуск

- запуск smoke набора: `pytest -m smoke -q`
- запуск регрессионного набора: `pytest -m regression -q`
- запуск полного набора: `pytest -q`
